# 2.0.0 — ORT/OpenVINO → MNN migration

Detailed notes on the 2.0.0 backend swap. CHANGELOG carries the headline;
this file carries the decisions, sizes, and gotchas.

## Why swap

The 1.x line was carrying ~400 MB of runtime dependencies to make
OpenVINO acceleration work reliably on Intel iGPUs (N100 etc.). The
acceleration was real, but the cost was:

- `onnxruntime-openvino` ~263 MB
- `intel-opencl-icd` + `ocl-icd-libopencl1` ~110 MB
- `libgomp1` and `supertonic` (ORT) on top
- 100+ lines of `ort.InferenceSession.__init__` monkey-patching to route
  OpenVINO only to `vector_estimator.onnx` and force plain CPU for the
  other three models (OV CPU plugin had a dynamic-shape Reshape bug
  that crashed on multilingual inputs unless we sidestepped it
  per-model)
- arch-conditional install ordering (install supertonic first, then
  swap ORT for the openvino variant) with a build-time assert
- `/dev/dri/renderD128` device + `video: true` group in `config.yaml`
- multi-shape warm-up sweep (English short, English long, Korean short,
  Korean long) to prime OV's dynamic-shape cache

`vra/supertonic-mnn` runs the same model through MNN (Alibaba's
lightweight inference engine). MNN ships statically-quantized int8/fp16
kernels for both x86 (AVX-VNNI) and ARM (NEON SDOT, FEAT_FP16), handles
dynamic shapes cleanly, and needs no iGPU access. None of the
workarounds above are needed.

## Image size impact

Roughly **~400 MB removed**, **~30 MB added** (MNN wheel + smaller MNN
model files). Net image shrink is large enough to be worth the major
version bump.

## API mapping (supertonic → supertonic-mnn)

| Old (`supertonic.TTS`)              | New (`supertonic_mnn.SupertonicTTS`)        |
| ----------------------------------- | -------------------------------------------- |
| `TTS(auto_download=True)`           | `SupertonicTTS(model_dir, precision, version)` |
| `tts.synthesize(text, voice_style=..., lang=..., steps=..., crop_silence_ms=..., speed=...)` | `tts.synthesize(text, voice="M1", lang="en", steps=5, speed=1.0)` |
| `tts.get_voice_style(voice_name=)`  | wrapped internally; voice passed as string |
| ORT providers via monkey-patched session | MNN config read from `config.json` on disk |

`crop_silence_ms` has no MNN equivalent and was removed from the option
schema. Sentence-edge silence is whatever the model produces natively;
in practice it's close to the previous default and inaudible at
conversational pace.

## supertonic-mnn version pin

PyPI's latest release at the time of the swap was `0.1.3`, which is
**v2-only** (5 languages) and `SupertonicTTS.synthesize()` doesn't take
a `lang=` kwarg. The `0.2.0` API (`version="v3"`, `lang="ko"`) lives on
git `main` only.

Until upstream cuts `0.2.0` to PyPI we install from a pinned commit:

```dockerfile
ARG SUPERTONIC_MNN_REF=ce5d7b25e2581293cd9d4ac0a462ac6e5e7d1cd0
RUN apt-get install -y --no-install-recommends git \
 && pip3 install "git+https://github.com/vra/supertonic-mnn.git@${SUPERTONIC_MNN_REF}" \
 && apt-get purge -y --auto-remove git
```

When `0.2.0` is on PyPI: drop the git install + `git` apt package, use
`pip install supertonic-mnn>=0.2.0`.

## MNN config.json seam

`supertonic_mnn.model.load_text_to_speech()` reads four MNN runtime
settings straight off disk and exposes none of them through Python:

```python
mnn_cfg["backend"] = mnn_backend_mapping[data['mnn_cfg_backend']]
mnn_cfg["thread_num"] = data['mnn_cfg_thread_num']
mnn_cfg["precision"] = data['mnn_cfg_precision']
mnn_cfg["memory"] = data['mnn_cfg_memory']
```

The HF model ships `thread_num=4, memory=low, precision=low (= fp16
runtime hint)`. To expose `threads` and `mnn_memory` as add-on options
without forking upstream, `engine._patch_mnn_config()` rewrites
`config.json` after `ensure_models()` downloads it. Idempotent — running
again with the same values is a no-op.

If `supertonic_mnn` ever grows a Python override for these, switch to
that and delete the patch.

## Auto precision heuristic

`engine._detect_best_precision()` reads `/proc/cpuinfo` once and picks:

- `int8` if x86 has `avx_vnni` or `avx512_vnni` (Intel 12th gen+,
  AMD Zen 4+, N100, Alder Lake-N)
- `int8` if ARM has `asimddp` (FEAT_DotProd / SDOT — Pi 5, Cortex-A75+)
- `fp16` if ARM has `asimdhp`/`fphp` but no DotProd
- `fp32` otherwise (Pi 4 Cortex-A72, A53, old x86 without VNNI)

Decision is logged. Manual override (`fp16`/`fp32`/`int8`) still works
for users who want to force a specific path.

The heuristic is conservative — it never picks something that's slower
than `fp32` on the host. On CPUs without hardware fp16, `fp16` is
emulated and identical-or-slower than `fp32`; we explicitly avoid
selecting it. On CPUs without VNNI/SDOT, `int8` adds quantize/dequantize
overhead that exceeds the int8 speedup; we avoid that too.

## CPU diagnostics

`engine.log_cpu_diagnostics()` runs at engine init and reads each CPU's
`scaling_governor`, `scaling_cur_freq`, `scaling_max_freq`,
`cpuinfo_max_freq` from sysfs. A loud warning fires when:

- `cpuinfo_max <= 1 GHz` on x86 (firmware locked turbo off — almost
  certainly intentional on a low-TDP mini PC, but worth surfacing)
- `governor == "powersave"` (caps at base clock on Intel — single most
  common cause of "RTF stuck around 2 on an N100")

If sysfs cpufreq is not visible from inside the container (some
configurations restrict this), the function logs one explanatory line
and moves on.

## TTFT

Per-request `TTFT` (time-to-first-audio-chunk) is logged once per
client request. Measured from the `Synthesize` / `SynthesizeStart`
event arriving at the handler to the first `AudioChunk` being written
to the socket. Includes lock-wait and PCM conversion — what the
listener actually perceives as "time until I hear sound". Subsequent
sentences in the same client request would log near-zero deltas and
are intentionally suppressed by clearing `_request_t0` after first use.

## Language UI

Schema in `config.yaml` is now English language names sorted
alphabetically (`Arabic`, `Bulgarian`, …, `Korean`, …, `Vietnamese`).
This makes the dropdown self-explanatory.

`const.resolve_language()` accepts either an English name or an ISO
code, so the engine still talks ISO codes internally and Wyoming
clients sending `language: "ko"` keep working.

HA does **not** translate `list(...)` schema values — there's no way to
show `한국어` in the dropdown for Korean users. The Korean translation
in `translations/ko.yaml` only carries option *names* and *descriptions*;
the dropdown is unavoidably English in every locale.

## Migration for existing users

Existing 1.x configs need three changes:

```yaml
# remove
provider: openvino_gpu
crop_silence: 300

# add
precision: auto
model_version: v3
mnn_memory: normal

# change
language: Korean      # was: language: ko
```

The HF model cache moves from `/data/.cache/supertonic3` (ONNX) to
`/data/.cache/supertonic-mnn` (MNN). First boot of 2.0.0 will
re-download the model into the new path; the old path can be deleted
to reclaim disk.

## Things measured / decided

- **Default precision**: `auto`. Best UX — picks the right thing on
  every CPU class without users having to know about VNNI/SDOT.
- **Default `mnn_memory`**: `normal`. `low` (the HF default) costs
  measurable speed; `high` was tested but only matters on tight RAM,
  which an HA add-on usually isn't.
- **Default `steps`**: bumped from 3 → 4 based on user feedback that
  3 sounds rough on Korean voices. 4 is the documented sweet spot in
  the upstream README too.
- **Default `model_version`**: `v3`. v2 is marginally faster but only
  supports 5 languages, so the default has to be v3 even though v2
  would speed up the small subset of users who only need en/ko/es/pt/fr.

## Things explicitly NOT done

- We do not bundle MNN models in the image. HF download on first start
  works fine and keeps the image small.
- We do not expose `mnn_cfg_backend` (cpu/opencl/vulkan/metal) as an
  option. Only `cpu` is supported in this add-on context — no other
  backend is available inside the HA add-on container without further
  device-passthrough work that would re-introduce the iGPU complexity
  we just removed.
- We do not implement a CUDA EP or any GPU backend. The point of 2.0.0
  is the removal of iGPU-related complexity. Users who need GPU
  inference should run a different TTS engine.
